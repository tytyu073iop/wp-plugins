## 3.9. Развёртывание и экспорт плагина

### 3.9.1 Подготовка плагина к экспорту

Перед распространением плагина необходимо выполнить сборку frontend-ресурсов. Процесс сборки использует `@wordpress/scripts` версии 30.24.0 (раздел 3.7). Команда `npm run build` запускает `wp-scripts build --webpack-copy-php`, что собирает исходные файлы из `src/` в директорию `build/`. Флаг `--webpack-copy-php` дополнительно копирует PHP-файлы, связанные с `block.json`, в директорию сборки.

Директория `build/` включена в `.gitignore` и не хранится в репозитории, поэтому после клонирования или переключения ветки необходима пересборка:

```sh
npm install
npm run build
```

Для создания дистрибутива рекомендуется исключить следующие файлы и директории, не требующиеся в production-окружении:

- `node_modules/`, `src/`, `package.json`, `package-lock.json` — зависимости и исходники не нужны на целевом сайте, `build/` уже содержит собранные файлы блоков;
- `.git/`, `.gitignore`, `.github/` — служебные файлы системы контроля версий;
- `.wp-env.json`, `.wordpress-org/` — конфигурация среды разработки.

Кастомный `webpack.config.js` отсутствует — `@wordpress/scripts` предоставляет стандартную конфигурацию webpack, достаточную для сборки Gutenberg-блоков.

Упаковка производится архиватором (zip) с именем `my-catalog.zip`, содержащим единственную корневую директорию `my-catalog/` со следующими элементами:

### 3.9.2 Развёртывание WordPress в Docker

Для развёртывания WordPress с плагином в Docker используется `docker-compose.yml`. Конфигурация включает сервисы `wordpress` (на основе официального образа `wordpress:6.8-php8.1-apache`) и `db` (MySQL 8.0). Плагин монтируется в `/var/www/html/wp-content/plugins/my-catalog/` через volume, что позволяет обновлять код без пересборки образа:

```yaml
services:
  wordpress:
    image: wordpress:6.8-php8.1-apache
    ports:
      - "8080:80"
    secrets:
      - db_password
      - auth_key
      - secure_auth_key
      - logged_in_key
      - nonce_key
      - auth_salt
      - secure_auth_salt
      - logged_in_salt
      - nonce_salt
    environment:
      WORDPRESS_DB_HOST: db
      WORDPRESS_DB_USER: wordpress
      WORDPRESS_DB_NAME: wordpress
      WORDPRESS_DB_PASSWORD_FILE: /run/secrets/db_password
      WORDPRESS_AUTH_KEY_FILE: /run/secrets/auth_key
      WORDPRESS_SECURE_AUTH_KEY_FILE: /run/secrets/secure_auth_key
      WORDPRESS_LOGGED_IN_KEY_FILE: /run/secrets/logged_in_key
      WORDPRESS_NONCE_KEY_FILE: /run/secrets/nonce_key
      WORDPRESS_AUTH_SALT_FILE: /run/secrets/auth_salt
      WORDPRESS_SECURE_AUTH_SALT_FILE: /run/secrets/secure_auth_salt
      WORDPRESS_LOGGED_IN_SALT_FILE: /run/secrets/logged_in_salt
      WORDPRESS_NONCE_SALT_FILE: /run/secrets/nonce_salt
    volumes:
      - ./my-catalog:/var/www/html/wp-content/plugins/my-catalog

  db:
    image: mysql:8.0
    secrets:
      - db_password
    environment:
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wordpress
      MYSQL_PASSWORD_FILE: /run/secrets/db_password
      MYSQL_RANDOM_ROOT_PASSWORD: "1"
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:

secrets:
  db_password:
    file: ./secrets/db_password.txt
  auth_key:
    file: ./secrets/auth_key.txt
  secure_auth_key:
    file: ./secrets/secure_auth_key.txt
  logged_in_key:
    file: ./secrets/logged_in_key.txt
  nonce_key:
    file: ./secrets/nonce_key.txt
  auth_salt:
    file: ./secrets/auth_salt.txt
  secure_auth_salt:
    file: ./secrets/secure_auth_salt.txt
  logged_in_salt:
    file: ./secrets/logged_in_salt.txt
  nonce_salt:
    file: ./secrets/nonce_salt.txt
```

Для управления чувствительными данными (пароли БД, ключи аутентификации WordPress) применяется механизм **Docker Secrets**. Секреты определяются в секции `secrets` Compose-файла и монтируются в контейнеры как файлы `/run/secrets/<имя>`. WordPress-образ поддерживает чтение секретов через переменные окружения с суффиксом `_FILE` — например, `WORDPRESS_DB_PASSWORD_FILE=/run/secrets/db_password` указывает образу прочитать пароль из указанного файла вместо передачи значения напрямую через переменную окружения. Аналогичный подход используется для `MYSQL_PASSWORD_FILE` в сервисе `db`.

Docker Secrets является нативным механизмом **Docker Swarm** — встроенного оркестратора Docker. Для корректной работы секретов в production-окружении Docker должен быть переведён в режим Swarm командой `docker swarm init`, а развёртывание стека выполняется через `docker stack deploy -c docker-compose.yml my-catalog` (вместо `docker-compose up`). В Swarm-режиме секреты хранятся в зашифрованном виде и передаются только тем узлам, на которых запущены сервисы, имеющие доступ к данному секрету.

Файлы секретов (`secrets/*.txt`) не включаются в репозиторий и генерируются локально. Для генерации валидных ключей аутентификации WordPress используется `wp-generate-salts`:

```sh
mkdir -p secrets
curl -s https://api.wordpress.org/secret-key/1.1/salt/ | \
  sed 's/define(/''/g; s/', '/''./g; s/' );//g' | \
  while IFS='.' read -r key value; do
    echo "$value" > "secrets/$(echo "$key" | tr '[:upper:]' '[:lower:]').txt"
  done
```

### 3.9.3 Загрузка и установка плагина в WordPress

Установка плагина в целевой WordPress выполняется через админ-панель: «Плагины» → «Добавить новый» → «Загрузить плагин», выбор zip-архива, затем «Установить» и «Активировать».

После активации выполняются следующие действия:

1. **Активационный хук** (`register_activation_hook`) вызывает функцию `my_catalog_activate()`, которая:
   - Регистрирует типы записей (`product`, `news`) и таксономии (`product_cat`, `product_tag`, `news_category`) через `My_Catalog_Core::register_content_types()`;
   - Сохраняет опцию `my_catalog_version` со значением `0.2.0`;
   - Вызывает `flush_rewrite_rules()` для обновления структуры постоянных ссылок.

2. **Инициализация** координатора `My_Catalog_Plugin::instance()`:
   - Создаёт экземпляры модулей `My_Catalog_Core`, `My_Catalog_Product_Table` и `My_Catalog_News_Carousel`;
   - Регистрирует Gutenberg-блоки через `register_block_type()` на хуке `init`;
   - Загружает текстовый домен для интернационализации на хуке `plugins_loaded`.

3. **REST endpoints** регистрируются на хуке `rest_api_init`:
   - `GET /my-catalog/v1/product-table` — серверная обработка DataTables;
   - `GET /my-catalog/v1/price-bounds` — границы цен для фильтра.

Настройка отображаемых колонок таблицы доступна в «Settings» → «My Catalog». При деактивации плагина вызывается `flush_rewrite_rules()` для очистки правил перезаписи, зарегистрированных во время активации.

### 3.9.4 Процедура uninstall.php

При удалении плагина через админ-панель («Плагины» → «Удалить») WordPress выполняет файл `uninstall.php`, который очищает только опцию версии:

```php
delete_option( 'my_catalog_version' );
```

Следующие данные **не удаляются** и остаются в базе данных WordPress:
- Записи типов `product` и `news`;
- Термины таксономий `product_cat`, `product_tag`, `news_category`;
- Мета-поля товаров (`_my_catalog_price`, `_my_catalog_sku`, `_my_catalog_stock_status`, `_my_catalog_weight`);
- Опция `my_catalog_product_table_columns` (настройки отображаемых колонок).

Такое поведение обусловлено тем, что `uninstall.php` содержит только одну операцию очистки. Записи и термины сохраняются в базе данных как «осиротевшие» данные, что позволяет восстановить функциональность плагина после повторной установки без потери контента.

### 3.9.5 Системные требования

Плагин имеет следующие минимальные системные требования:

- **WordPress:** 6.5+ (проверка `Requires at least: 6.5` в заголовке плагина);
- **PHP:** 7.4+ (проверка `Requires PHP: 7.4`);
- **Node.js:** 18+ (требуется для сборки frontend-ресурсов через `@wordpress/scripts` версии 30.24.0, который требует Node ≥ 18).

На целевом сайте Node.js не требуется — сборка производится на стороне разработчика перед распространением.
