# Глава 3. Реализация плагина

## 3.1. Экранные формы и пользовательский интерфейс

### 3.1.1 Административная страница настроек
Первое, с чем сталкивается автор после активации плагина на своём сайте, — это страница настроек «My Catalog» в меню «Settings» (Рисунок 3.1). На этой странице расположен набор флажков для выбора колонок, отображаемых по умолчанию в таблице товаров. Каждая колонка соответствует одному из мета-полей товара (цена, артикул, наличие, вес). Сохранение настроек осуществляется через стандартный механизм Settings API WordPress.

### 3.1.2 Фронтенд: блок таблицы товаров
На фронтенде автор размещает Gutenberg-блок «Product Table» в произвольной записи или на странице (Рисунок 3.2). Блок поддерживает атрибуты `limit`, `category`, `tag`, `columns`, `search`, `tableId`, `emptyMessage`. Таблица рендерится на стороне сервера с пустым телом, после чего JavaScript DataTables загружает данные через REST API.

### 3.1.3 Фронтенд: Gutenberg-блок «Карусель новостей»
В редакторе блоков автору доступен блок «News Carousel» (Рисунок 3.3). В панели инспектора можно настроить количество элементов, категорию новостей, автовоспроизведение и задержку. В режиме редактирования блок отображается через `ServerSideRender`, а на фронтенде — Swiper-карусель с поддержкой пагинации и навигационных стрелок.

### 3.1.4 Фронтенд: блок фильтров товаров
Блок «Product Filters» позволяет разместить фильтры по категории и цене отдельно от таблицы. Фильтры связываются с таблицей через атрибут `target` (ID таблицы). Если ID не указан, фильтры управляют всеми таблицами на странице.

## 3.2. Регистрация произвольных типов записей и таксономий

### 3.2.1 Тип записи `product` (slug: products)
Тип записи `product` регистрируется через `register_post_type()` с поддержкой REST API (`show_in_rest => true`), миниатюр, отрывков и архива. URL перезаписывается как `/products/`.

### 3.2.2 Тип записи `news` (slug: news)
Тип записи `news` регистрируется аналогично, с slug `/news/`. Иконка в админ-панели — `dashicons-megaphone`.

### 3.2.3 Таксономии товаров
- `product_cat` — иерархическая, для категорий товаров.
- `product_tag` — неиерархическая, для атрибутов товаров. В админ-панели отображается как «Product Attributes» (Приложение А).

### 3.2.4 Таксономии новостей
- `news_category` — иерархическая, для категорий новостей.

## 3.3. Реализация мета-боксов

### 3.3.1 Мета-бокс «Product Details»
Для типа записи `product` добавлен мета-бокс с полями: цена (`_my_catalog_price`), артикул (`_my_catalog_sku`), наличие (`_my_catalog_stock_status`), вес (`_my_catalog_weight`). Все мета-ключи имеют префикс `_my_catalog_`, что делает их защищёнными (скрытыми из стандартного UI произвольных полей).

### 3.3.2 Мета-бокс «News Details»
Не реализован. Новости используют только стандартные поля WordPress (заголовок, содержание, изображение).

### 3.3.3 Сохранение и валидация
Сохранение мета-полей выполняется через хук `save_post_product`. Валидация включает проверку nonce, защиту от автосохранений и ревизий, санитацию данных (десятичные числа, ключи).

## 3.4. Хранение данных

### 3.4.1 Post Meta
Данные товаров хранятся в стандартной таблице `wp_postmeta`. Ключи используют префикс `_my_catalog_`. Для хранения используются типы `string` (цена, артикул, вес) и `string` с фиксированным набором значений (наличие).

### 3.4.2 Options API
Настройки отображаемых колонок таблицы хранятся через `register_setting()` с типом `array`. Опция `my_catalog_product_table_columns` использует санитаризацию через `sanitize_columns()`, отсекающую невалидные ключи.

### 3.4.3 REST API endpoints
Для обеспечения работы DataTables на фронтенде реализованы два endpoint'а (Приложение Б):
- `GET my-catalog/v1/product-table` — основной endpoint, принимает параметры поиска, сортировки, фильтрации (по категории, тегу, цене) и пагинации (DataTables protocol: `draw`, `start`, `length`). Обработчик запроса строит два `WP_Query`: один для общего количества записей, второй — с применением всех фильтров. Поиск выполняется кастомными `posts_clauses` по заголовку, содержанию, мета-полям и названиям терминов таксономий.
- `GET my-catalog/v1/price-bounds` — возвращает минимальную и максимальную цену среди товаров для инициализации ценового диапазона в фильтрах.

## 3.5. Реализация Gutenberg-блоков

Плагин регистрирует три динамических Gutenberg-блока, каждый со своим файлом `block.json`. Два блока (`product-table`, `product-filters`) заменяют ранее существовавшие шорткоды. Третий блок (`news-carousel`) был изначально реализован как блок.

### 3.5.1 Регистрация блоков
Блоки регистрируются через `register_block_type()` в координаторе `My_Catalog_Plugin`:
- `my-catalog/product-table` — `render_callback` на `My_Catalog_Product_Table::render_table_block()`
- `my-catalog/product-filters` — `render_callback` на `My_Catalog_Product_Table::render_filters_block()`
- `my-catalog/news-carousel` — `render_callback` на `My_Catalog_News_Carousel::render_block()`

### 3.5.2 Атрибуты и генерация HTML
Атрибуты каждого блока описаны в `block.json`. Метод `render_table_block()` парсит атрибуты, формирует JSON-конфигурацию с URL REST, базовыми фильтрами и списком колонок, после чего генерирует контейнер `<div>` с `data-config`. В теле таблицы выводится строка с сообщением об отсутствии товаров — реальные данные загружаются через AJAX.

### 3.5.3 Фильтры товаров
Блок `my-catalog/product-filters` рендерит выпадающий список категорий и диапазон цен (с двумя бегунками `input[type="range"]` и числовыми полями). Фильтры взаимодействуют с DataTables через JavaScript, обновляя запрос к REST endpoint.

### 3.5.4 Inline-шорткоды
Шорткоды `[my_catalog_price]` и `[my_catalog_stock]` сохранены для обратной совместимости и предназначены для вывода отдельных полей текущего товара внутри произвольного содержимого.

## 3.6. Реализация динамического Gutenberg-блока (News Carousel)

### 3.6.1 Регистрация блока
Блок `my-catalog/news-carousel` регистрируется через `register_block_type()` с `render_callback` на `My_Catalog_News_Carousel::render_block()`. Используется API v3, `save()` возвращает `null` (динамический блок).

### 3.6.2 Серверный рендеринг
В редакторе блок отображается через `ServerSideRender`. PHP-метод `render_block()` принимает атрибуты блока, маппит camelCase-ключи в snake_case и передаёт в `render()`, который выполняет `WP_Query` для получения новостей и формирует Swiper-карусель.

### 3.6.3 Переопределение шаблона слайда
Тема может переопределить шаблон слайда, разместив файл `my-catalog/news-slide.php` или `my-catalog-news-slide.php`. `locate_template()` проверяет их в порядке приоритета. Встроенный шаблон (Приложение Г) выводит медиа, заголовок, отрывок и ссылку «Read More» на страницу новости.

### 3.6.4 Блок «Product Filters»
Аналогичным образом регистрируется блок `my-catalog/product-filters` с атрибутами `target`, `showCategory`, `showPrice`.

## 3.7. Сборка frontend-ресурсов

### 3.7.1 Webpack через `@wordpress/scripts`
Исходные файлы блоков (`src/index.js`, `src/editor.scss`, `src/style.scss`) собираются в `build/` командой `npm run build`. Флаг `--webpack-copy-php` копирует PHP-файлы в `build/` при необходимости.

### 3.7.2 Подключение библиотек через CDN
DataTables 1.13.8 и Swiper 11.1.4 подключаются с CDN. Для каждой библиотеки регистрируются зависимости: jQuery для DataTables, отсутствие зависимостей для Swiper. CSS и JS подключаются раздельно.

## 3.8. Архитектура плагина и принципы проектирования

### 3.8.1 Структура классов
Плагин организован по модульному принципу:
- `My_Catalog_Plugin` — координатор (синглтон), создаёт три модуля, регистрирует блоки и текстовый домен.
- `My_Catalog_Core` — отвечает за типы записей, таксономии и мета-боксы.
- `My_Catalog_Product_Table` — блоки таблицы и фильтров, REST API, страница настроек.
- `My_Catalog_News_Carousel` — рендеринг карусели новостей.

### 3.8.2 Принципы SOLID
**S (Single Responsibility):** Каждый класс отвечает за одну функциональную область: `My_Catalog_Core` — только контентные типы, `My_Catalog_Product_Table` — только таблицу товаров.

**O (Open/Closed):** Добавление новых колонок таблицы происходит через расширение массива в `get_available_columns()` без изменения логики рендеринга.

**L (Liskov Substitution):** Санитаризация колонок `sanitize_columns()` гарантирует, что любые переданные данные приводятся к валидному набору, что позволяет подменять источник данных без поломки логики.

**I (Interface Segregation):** Каждый публичный метод класса имеет узкую ответственность; методы группируются по модулям.

**D (Dependency Inversion):** Модули зависят от WordPress API (абстракции хуков и фильтров), а не от конкретных реализаций друг друга.

### 3.8.3 Шаблоны проектирования
**Singleton:** `My_Catalog_Plugin::instance()` гарантирует единственный экземпляр координатора плагина (Приложение Д).

**Adapter:** Регистрация `render_callback` в `register_block_type()` адаптирует PHP-метод класса под интерфейс Gutenberg.

**Strategy:** Метод `run_products_query()` принимает массив параметров, определяющих стратегию построения запроса (с фильтрацией, без, с поиском); конкретная логика инкапсулирована внутри.

## 3.9. Развёртывание и экспорт плагина

### 3.9.1 Подготовка плагина к экспорту
Перед распространением плагина необходимо выполнить `npm run build` для сборки frontend-ресурсов в директорию `build/`. Для создания дистрибутива рекомендуется исключить следующие файлы и директории, не требующиеся в production-окружении:
- `node_modules/`, `src/`, `package.json`, `package-lock.json`, `webpack.config.js`
- `.git/`, `.gitignore`, `.github/`
- `tests/`, `.wp-env.json`

Упаковка производится архиватором (zip) с именем `my-catalog.zip`, содержащим единственную корневую директорию `my-catalog/` с файлами плагина.

### 3.9.2 Развёртывание WordPress в Docker
Для развёртывания WordPress с плагином в Docker используется `docker-compose.yml`. Конфигурация включает сервисы `wordpress` (на основе официального образа `wordpress:6.8-php8.1-apache`) и `db` (MySQL). Плагин монтируется в `/var/www/html/wp-content/plugins/my-catalog/` через volume.

Для управления чувствительными данными (пароли БД, ключи аутентификации) применяется механизм **Docker Secrets**. Секреты определяются в Compose-файле и монтируются в сервисы как файлы `/run/secrets/<имя>`. WordPress-образ поддерживает чтение секретов через переменные окружения `WORDPRESS_DB_PASSWORD_FILE` и аналогичные.

### 3.9.3 Загрузка и установка плагина в WordPress
Установка плагина в целевой WordPress выполняется одним из следующих способов:
- **Через админ-панель:** «Плагины» → «Добавить новый» → «Загрузить плагин», выбор zip-архива, активация.

После активации плагин регистрирует типы записей, таксономии и REST endpoints. Настройка отображаемых колонок таблицы доступна в «Settings» → «My Catalog».

### 3.9.4 Системные требования
- WordPress 6.5+
- PHP 7.4+
- Node.js 18+ (для сборки)

### 3.9.5 Первоначальная настройка
После клонирования репозитория необходимо выполнить `npm install`, затем `npm run build` (build-директория gitignored). Для локальной разработки используется `wp-env` с WordPress 6.8.

### 3.9.6 uninstall.php
При удалении плагина удаляется только опция версии (`my_catalog_version`). Записи, термины и мета-поля остаются в базе данных.

## 3.10. Пример использования

### 3.10.1 Создание товаров автором
Автор создаёт несколько товаров через «Products» в админ-панели: добавляет название, описание, изображение (через миниатюру), указывает цену, артикул, категорию, статус наличия. Категории и атрибуты создаются заранее через соответствующие таксономии.

### 3.10.2 Размещение таблицы на странице (Начальный вид)
Автор добавляет на страницу в редакторе Gutenberg блок «Product Table» с параметрами `limit="5"`, `columns="image,title,price,category,stock"`. На фронтенде отображается пустая таблица с надписью «No products found» — данные загружаются асинхронно через REST API.

### 3.10.3 Работа с таблицей
Посетитель сайта может выполнять поиск, сортировку по заголовку и цене, фильтрацию по категориям и диапазону цен (при наличии блока фильтров). DataTables обрабатывает взаимодействие на стороне клиента, отправляя AJAX-запросы к `GET /my-catalog/v1/product-table`.

### 3.10.4 Карусель новостей (Итоговый вид)
Автор добавляет блок «News Carousel» в запись. В панели настроек выбирается лимит (6), категория новостей («Все»), включается автовоспроизведение с задержкой 5000 мс. На фронтенде отображается Swiper-карусель с навигацией (стрелки и пагинация) и слайдами новостей (изображение, заголовок, отрывок, кнопка «Read More»).
